Understanding Formal Verification

What does “mathematically proven” actually mean?

Formal verification isn’t magic — it’s a well-understood branch of computer science that’s been used in aerospace, chip design, and nuclear systems for decades. Here’s how to think about it.

🔍
Like spell-check, but for logic

Spell-Check for Code

Spell-check doesn’t read your essay — it checks every word against rules. AEON doesn’t run your code — it checks every possible execution against mathematical rules. If it says “correct,” it means every path is safe.

💻
Like an X-ray, not a spot check

X-Ray for Software

Testing checks a few inputs and hopes for the best. AEON examines the structure of your code — like an X-ray that sees every bone, not just the ones that hurt. It finds bugs that no amount of testing would catch.

🔐
Like a mathematical proof, not a guess

Proof, Not Probability

When a mathematician proves the Pythagorean theorem, it’s true for every right triangle — not just the ones they measured. AEON proves your code works for every possible input, not just the test cases you wrote.

What AEON Checks

30+ verification engines

Each engine targets a specific class of bugs. Plain English below — toggle to see the computer science.

● Correctness
🔎

Symbolic Execution

Tries every possible input to your function and tells you exactly which ones cause crashes. Gives you the specific value that breaks your code.

Path-sensitive analysis exploring all feasible execution paths. Variables are symbolic; branches fork into two states. Path conditions checked by Z3 for feasibility. Generates concrete counterexamples.

King 1976 · KLEE/OSDI 2008
🔐

Abstract Interpretation

Tracks the range of every variable (e.g., “this number is between 0 and 100”) and catches overflow, division by zero, and out-of-bounds errors.

Static analysis via Galois connections between concrete and abstract domains. Interval domain with threshold widening, sign domain Hasse lattice, congruence domain via CRT. Fixpoint computation uses widening (∇) and narrowing (Δ).

Cousot & Cousot, POPL 1977
📜

Hoare Logic

Checks your function’s promises: “If you give me valid input, I guarantee valid output.” Mechanically verifies that every contract is honored.

Weakest precondition calculus via wp(S,Q) substitution, composition, and conditional rules. Verification conditions discharged to Z3 SMT solver. Loop invariants inferred using Houdini’s algorithm.

Hoare 1969 · Dijkstra 1975 · Flanagan & Leino 2001
🎯

Liquid Types

Attaches conditions to types: not just “this is a number” but “this is a number greater than zero.” The compiler then proves these conditions are always met.

Refinement types annotate base types with logical predicates verified by SMT. Inference via predicate abstraction over qualifier templates, solving for strongest conjunction via CEGAR — a fixpoint in the qualifier conjunction lattice.

Rondon, Kawaguchi, Jhala — PLDI 2008
🧮

Dependent Types

Types that carry data: instead of “list of numbers,” you can say “list of exactly 5 numbers.” Your program literally becomes a mathematical proof of its own correctness.

Full Pi types where return types depend on input values. Via Curry-Howard, types are propositions and programs are proofs. Bidirectional type checking, beta/eta normalization, universe hierarchy Type₀ : Type₁ : …

Martin-Löf 1984 · Coquand & Huet 1988
🚫

Dead Code Detection

Finds code that can never run: statements after a return, unused variables, parameters nobody reads, and if/else branches that do the same thing. Keeps your codebase clean.

Control flow analysis identifies unreachable code after terminators. Def-use chains detect unused definitions. Branch equivalence via structural comparison. Self-assignment and redundant condition detection.

Allen 1970 · Knoop, Rüthing & Steffen 1994
📈

Abstract Refinement Types

Goes beyond basic refinement types by parameterizing over abstract predicates. Enables generic, reusable verified components where the proof is also polymorphic.

Parametric refinement types abstract over predicates. Enables type-polymorphic verification: a single sort implementation verified once works for all orderings. Solved via predicate abstraction with CEGAR.

Vazou et al., ESOP 2013 · ICFP 2014
📐

Gradual Typing Verification

Handles codebases that mix typed and untyped code. Verifies what it can statically and inserts runtime checks for the rest — no all-or-nothing tradeoff.

Consistent subtyping with the unknown type (?). Static guarantees degrade gracefully at typed/untyped boundaries. Runtime cast insertion for dynamic regions. Blame tracking identifies the source of type failures.

Siek & Taha 2006 · Bader et al. 2018
🔭

Craig Interpolation / CEGAR

When a proof fails, automatically figures out what extra information is needed and refines the analysis. Keeps getting smarter until it either proves or finds a real bug.

Counterexample-guided abstraction refinement (CEGAR). Uses Craig interpolants from infeasible error traces to discover new predicates. Iteratively refines the abstract domain until proof or concrete counterexample.

McMillan, CAV 2003 · Henzinger et al., POPL 2004
📊

Complexity Analysis

Automatically computes how much time and memory your function uses as a function of input size. Catches accidentally quadratic algorithms and memory leaks.

Resource-aware type system (RAML). Annotates types with potential functions. Derives polynomial bounds on heap, stack, and time usage. Amortized analysis via the potential method.

Hoffmann et al., TOPLAS 2012 · Gulwani et al., POPL 2009
🔀

Relational Verification

Compares two versions of a function and proves they behave the same. Essential for safe refactoring, compiler optimization validation, and security (non-interference).

2-safety via product programs. Self-composition for non-interference. Relational Hoare logic for comparing two program executions. Proves equivalence, refinement, and relative correctness.

Barthe et al., FM 2011 · Benton, POPL 2004
● Safety
⏳

Termination Analysis

Proves your functions will always finish running — no infinite loops, no hangs, no frozen programs. Uses a clever mathematical trick from Ramsey theory.

Decides termination using Ramsey’s theorem. Builds size-change graphs at each call site, computes transitive closure under graph composition, and verifies strict decrease exists for every idempotent self-loop SCG. Sound and complete for SCT.

Lee, Jones, Ben-Amram — POPL 2001
🚧

Separation Logic

Prevents memory bugs: using freed memory, freeing memory twice, or forgetting to free memory at all. This is the math behind Facebook’s Infer tool.

Heap safety via spatial reasoning. Separating conjunction (P * Q) splits heap into disjoint parts. Frame rule enables modular verification. Bi-abduction (Calcagno et al. 2011) for automatic pre/postcondition inference.

Reynolds 2002 · O’Hearn 2019 · Calcagno et al. 2011
🔄

Concurrency

Finds race conditions (two threads changing the same data at once) and deadlocks (threads waiting for each other forever). Critical for multi-threaded servers.

Lockset analysis (Eraser/Savage et al. 1997) tracks locks held during shared access. Happens-before (Lamport 1978) partial order on concurrent events. Cycle detection in lock-order graphs for deadlock prevention.

Owicki & Gries 1976 · Savage et al. 1997
🔬

Model Checking

Exhaustively checks every possible state your program can be in, up to a limit. Like testing every combination of inputs, but done mathematically.

Bounded state-space exploration. Program unrolled to bound k, encoded as SAT/SMT formula. Temporal logic (CTL): AG P (safety), AF P (liveness), EF P (reachability). Counterexample traces for assertion violations.

Clarke, Emerson, Sistla 1986 · Biere et al. 1999
🗈

Shape Analysis

Verifies that your linked lists don’t have cycles, your trees are balanced, and you never follow a null pointer. Essential for data-structure-heavy code.

Verification of linked structures via 3-valued logic (Kleene). Canonical abstraction merges nodes into summary nodes by predicates. Shape predicates: reach(x,y), cycle(x), shared(x), sorted(x).

Sagiv, Reps, Wilhelm — TOPLAS 2002
🔢

Numeric Safety

Catches math bugs: integer overflow when numbers get too big, dividing by zero, comparing floats with == (unreliable), and mixing signed/unsigned integers. Silent data corruption, prevented.

Abstract numeric domain analysis. Tracks integer bounds for overflow/underflow. Detects lossy float-to-int coercions, NaN propagation, unsigned subtraction wrap, and signed/unsigned arithmetic mismatch.

Miné 2004 · Dietz et al. 2012
🔢

Null Safety

Tracks which variables might be null and warns you before you use them. Catches the #1 runtime error in most languages — the dreaded NullPointerException — before it happens.

Null-state tracking with narrowing through control flow. Three states: DEFINITELY_NULL, DEFINITELY_NOT_NULL, MAYBE_NULL. Null guard analysis narrows state after if (x != null). Detects unsafe optional unwrap.

Fähndrich & Leino 2003
🔗

Linear / Affine Resources

Ensures every resource (file handle, connection, lock) is used exactly once — never leaked, never double-freed. The compiler enforces it, not the programmer.

Linear type system based on Girard’s linear logic (1987). Resources must be consumed exactly once (linear) or at most once (affine). Detects resource leaks, double-free, and use-after-close at compile time.

Girard 1987 · Hofmann & Jost, ESOP 2003
🔐

Type-State Analysis

Tracks the state of objects through their lifecycle (e.g., a file must be opened before reading and closed after). Catches protocol violations at compile time.

Augments type system with object states and valid transitions. State machine per type tracks operations valid in each state. Prevents out-of-order API calls and resource protocol violations.

Strom & Yemini 1986 · DeLine & Fähndrich, ECOOP 2004
🎲

Probabilistic Analysis

Reasons about programs that use randomness — sampling, machine learning, simulations. Proves probabilistic bounds like “this classifier is at least 95% accurate.”

Probabilistic program semantics via measure-theoretic foundations. Computes distribution bounds, expectation analysis, and concentration inequalities. Supports Monte Carlo verification and Bayesian inference.

Kozen 1981 · Gordon et al. 2014
● Reliability
📄

API Contract Verification

Checks that you validate inputs before using them, check return values from operations that can fail, and don’t use resources (files, connections) after closing them.

Tracks resource lifecycle (OPEN/CLOSED) with protocol verification. Detects unchecked returns from fallible operations, missing parameter validation, and use-after-close violations.

Acharya et al. 2007 · Engler et al. 2001
⚠

Error Handling

Finds swallowed errors (empty catch blocks that hide failures), missing error handling for operations that can fail, and resources that leak when something goes wrong.

Exception flow analysis tracking error propagation paths. Detects empty catch handlers, catch-all without re-throw, missing error handling for fallible operations, and resource cleanup gaps on error paths.

Robillard & Murphy 2000 · Weimer & Necula 2004
● Security
🔒

Information Flow

Proves that secret data (passwords, API keys, personal info) can never leak to public outputs. Not by searching for patterns — by mathematically proving the data flow is safe.

Security type system proving noninterference — secret inputs cannot influence public outputs. Security lattice: PUBLIC ≤ INTERNAL ≤ SECRET ≤ TOP_SECRET. Detects explicit, implicit, and termination channel leaks.

Volpano, Smith, Irvine 1996
🕷

Taint Analysis

Tracks user input from the moment it enters your app to where it’s used. If untrusted data reaches a database query or HTML output without sanitization, AEON catches it.

Injection detection by tracking untrusted data. Sources: HTTP params, user input, file reads, env vars. Sinks: SQL queries, HTML output, OS commands. Propagation through assignments, operations, and calls. Detects SQLi, XSS, SSRF, path traversal.

Schwartz et al. 2010 · Tripp et al. PLDI 2009 · Arzt et al. 2014
🔐

Differential Privacy

Proves that your data analysis algorithms protect individual privacy. Guarantees that adding or removing one person’s data doesn’t significantly change the output.

Verifies (ε, δ)-differential privacy via sensitivity analysis on queries. Tracks privacy budget composition. Detects unbounded sensitivity, missing noise injection, and privacy budget exhaustion.

Reed & Pierce, ICFP 2010 · Gaboardi et al., POPL 2013
● Architecture
⚡

Algebraic Effects

Tracks what side effects your code has (database reads, network calls, file writes) and ensures they’re declared and controlled. No hidden surprises.

Effects form a bounded lattice with row-polymorphic typing. Effect rows with row variable polymorphism, commutativity analysis for auto-parallelization, handler semantics via delimited continuations.

Plotkin & Pretnar, ESOP 2009 · Leijen, POPL 2017
📐

Category Semantics

Gives your code a precise mathematical meaning. Pure functions, side effects, and data types each have a rigorous definition — which means the compiler can reason about them perfectly.

Denotational semantics: pure functions → morphisms in a Cartesian Closed Category (CCC); task functions → Kleisli morphisms in a graded monad TE; data types → initial algebras of polynomial functors.

Moggi 1991
🛡

Certified Compilation

Proves that the compiled code does exactly what your source code says. No compiler bugs, no optimization surprises. Inspired by the CompCert verified C compiler.

CompCert-style simulation proofs verify each compiler pass preserves semantics. Forward simulation, invariant tracking, translation validation, and compositional pass-by-pass correctness.

Leroy, CACM 2009
💬

Session Types

Verifies that communicating components follow their protocol correctly. If a client must send a request before receiving a response, the compiler enforces it.

Binary session types via the Curry-Howard correspondence with linear logic. Duality ensures each send has a matching receive. Multiparty session types extend to n-party protocols. Detects deadlocks and protocol violations.

Honda et al., POPL 2008 · Wadler, ICFP 2012

Get Started

Up and running in 30 seconds

Choose your platform:

Install AEON via pip and verify your first file:

# Install from PyPI
pip install aeon-lang

# Verify any file (auto-detects language)
aeon check your_code.py --deep-verify
aeon check Main.java --deep-verify
aeon check server.go --deep-verify

Install via Homebrew (macOS & Linux):

brew tap aeon-lang/tap
brew install aeon
aeon check your_code.rs --deep-verify

One-line install for Linux & macOS:

curl -fsSL https://raw.githubusercontent.com/aeon-lang/aeon/main/install.sh | bash

Install on Windows via PowerShell or Scoop:

# PowerShell
irm https://raw.githubusercontent.com/aeon-lang/aeon/main/install.ps1 | iex

# Scoop
scoop bucket add aeon https://github.com/aeon-lang/scoop-aeon
scoop install aeon

Run AEON in Docker — no installation needed:

docker run -v $(pwd):/workspace aeon-lang/aeon \
  aeon check /workspace/your_code.py --deep-verify

For JS/TS developers:

npx aeon-verify check app.ts --deep-verify

Install the AEON Verify VS Code extension:

cd vscode-extension
npm install && npm run compile
npx vsce package
code --install-extension aeon-verify-0.5.0.vsix

Add AEON verification to your CI pipeline:

# .github/workflows/verify.yml
name: AEON Verify
on: [push, pull_request]
jobs:
  verify:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with:
          python-version: '3.11'
      - run: pip install aeon-lang
      - run: aeon scan src/ --deep-verify --format sarif
