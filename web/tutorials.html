Your First 5 Minutes with AEON

Beginner 5 min Python

Install AEON, verify your first file, find a bug, fix it, and re-verify. The complete beginner workflow.

Step 1: Install AEON

pip install aeon-lang

Step 2: Create a file with a bug

Create example.py:

def average(numbers: list, count: int) -> float:
    return sum(numbers) / count

Step 3: Run AEON

aeon check example.py --deep-verify

AEON finds the bug:

✗ 2 bug(s) found
  ⚠ Division by zero possible (line 2)
    Engine: Abstract Interpretation (Cousot 1977)
    divisor_range: (-∞, +∞) includes 0
  ⚠ Symbolic Execution: division by zero reachable
    Counterexample: count = 0
    Fix: add requires count != 0

Step 4: Fix the bug with a contract

def average(numbers: list, count: int) -> float:
    """
    Requires: count != 0
    """
    return sum(numbers) / count

Step 5: Re-verify

aeon check example.py --deep-verify

✓ VERIFIED: 1 function, 0 classes — no bugs found

That's it. AEON mathematically proved your function is correct for all possible inputs where count != 0.

Adding Contracts to an Existing Python Project

Beginner 10 min Python

You don't need to rewrite your code. AEON reads contracts from Python docstrings. This tutorial shows how to progressively add contracts to an existing codebase.

Step 1: Start with baseline mode

Create a baseline of current issues so you only see new bugs going forward:

aeon scan src/ --deep-verify --create-baseline --baseline .aeon-baseline.json

Step 2: Add contracts to critical functions first

Start with payment processing, auth, and data access functions:

def process_payment(amount: int, balance: int) -> int:
    """
    Requires: amount > 0
    Requires: balance >= amount
    Ensures:  result == balance - amount
    Ensures:  result >= 0
    """
    return balance - amount

Step 3: Run with baseline to see only new issues

aeon scan src/ --deep-verify --baseline .aeon-baseline.json

Only newly introduced bugs are reported. Existing legacy issues are suppressed until you're ready to fix them.

Step 4: Add to CI/CD

Gate PRs on AEON verification so no new bugs slip in:

# .github/workflows/verify.yml
name: AEON Verify
on: [pull_request]
jobs:
  verify:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with: { python-version: '3.11' }
      - run: pip install aeon-lang
      - run: aeon scan src/ --deep-verify --baseline .aeon-baseline.json

Tip: Start with --verify (contracts only) instead of --deep-verify for a gentler rollout. Gradually enable more engines as your team adds contracts.

Setting Up AEON in CI/CD (GitHub Actions)

Intermediate 10 min YAML / CI

Integrate AEON into your GitHub Actions pipeline with SARIF output, code scanning annotations, and pass/fail gating.

Step 1: Basic workflow

# .github/workflows/aeon-verify.yml
name: AEON Verify
on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

jobs:
  verify:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - uses: actions/setup-python@v5
        with:
          python-version: '3.11'

      - name: Install AEON
        run: pip install aeon-lang

      - name: Run AEON verification
        run: aeon scan src/ --deep-verify --format sarif > aeon-results.sarif
        continue-on-error: true

      - name: Upload SARIF to GitHub Code Scanning
        uses: github/codeql-action/upload-sarif@v3
        with:
          sarif_file: aeon-results.sarif

      - name: Fail on bugs
        run: aeon scan src/ --deep-verify

Step 2: Add caching for faster builds

      - uses: actions/cache@v4
        with:
          path: ~/.cache/pip
          key: ${{ runner.os }}-pip-aeon
          restore-keys: ${{ runner.os }}-pip-

Step 3: Use baseline for incremental adoption

Commit your baseline file and only flag new issues in PRs:

      - name: Run with baseline
        run: aeon scan src/ --deep-verify --baseline .aeon-baseline.json

What you get

  • AEON findings appear as code annotations directly in your PR
  • GitHub Security tab shows all findings with SARIF
  • PRs are blocked if new bugs are introduced
  • Baseline mode allows gradual adoption without noise

Securing a Flask API with Taint Analysis

Intermediate 15 min Python / Flask

Use AEON's taint analysis engine to find and fix SQL injection, XSS, and information leaks in a Flask application.

Step 1: The vulnerable code

from flask import Flask, request
import sqlite3

app = Flask(__name__)

@app.route('/users')
def get_user():
    username = request.args.get('name')  # TAINT SOURCE
    conn = sqlite3.connect('app.db')
    # BUG: SQL injection — user input flows to query
    result = conn.execute(f"SELECT * FROM users WHERE name = '{username}'")
    return str(result.fetchall())

Step 2: Run AEON taint analysis

aeon check app.py --taint

Output:

✗ 1 bug found
  ⚠ AEON-E007: SQL injection
    Engine: Taint Analysis (Schwartz et al. 2010)
    Source: request.args.get('name') [line 8]
    Sink: conn.execute(...) [line 10]
    Path: username → f-string → conn.execute()
    Fix: use parameterized query

Step 3: Fix with parameterized queries

@app.route('/users')
def get_user():
    username = request.args.get('name')
    conn = sqlite3.connect('app.db')
    # SAFE: parameterized query neutralizes taint
    result = conn.execute("SELECT * FROM users WHERE name = ?", (username,))
    return str(result.fetchall())

Step 4: Re-verify

aeon check app.py --taint
✓ VERIFIED: no taint violations found

Step 5: Add custom sources/sinks

Configure additional sources and sinks for your codebase:

# .aeonrc.yml
taint:
  custom_sources:
    - "flask.request.args"
    - "flask.request.form"
    - "flask.request.json"
  custom_sinks:
    - "subprocess.call"
    - "os.system"
    - "eval"
  custom_sanitizers:
    - "bleach.clean"
    - "markupsafe.escape"

Verifying a Payment System End-to-End

Advanced 15 min Python

Build a payment processing module with full formal contracts. AEON proves no overflow, no negative balances, correct fee calculations, and no information leaks — for all possible inputs.

Step 1: Define the data model

from dataclasses import dataclass

@dataclass
class Account:
    id: str
    balance: int  # cents
    tier: str     # 'standard' | 'premium' | 'enterprise'

Step 2: Write functions with contracts

def calculate_fee(amount: int, tier: str) -> int:
    """
    Requires: amount > 0
    Ensures:  result >= 0
    Ensures:  result <= amount
    """
    rates = {'standard': 29, 'premium': 25, 'enterprise': 19}
    rate = rates.get(tier, 29)  # basis points (29 = 0.29%)
    return (amount * rate) // 10000

def transfer(sender: Account, receiver: Account, amount: int) -> tuple:
    """
    Requires: amount > 0
    Requires: sender.balance >= amount
    Ensures:  result[0].balance == sender.balance - amount
    Ensures:  result[1].balance == receiver.balance + amount - calculate_fee(amount, sender.tier)
    """
    fee = calculate_fee(amount, sender.tier)
    new_sender = Account(sender.id, sender.balance - amount, sender.tier)
    new_receiver = Account(receiver.id, receiver.balance + amount - fee, receiver.tier)
    return (new_sender, new_receiver)

Step 3: Verify with all engines

aeon check payment.py --deep-verify

✓ VERIFIED: 2 functions, 1 class — no bugs found

AEON has now mathematically proven:

  • No overflow in fee calculation (Abstract Interpretation)
  • No negative balances possible (Hoare Logic)
  • Fee is always non-negative and bounded (Symbolic Execution)
  • Transfer conserves money (Hoare Logic — ensures clause)
  • All branches terminate (Size-Change Termination)
