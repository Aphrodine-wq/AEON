Engine-by-Engine Research Map

Each of AEON’s 40 engines traces directly to foundational publications. Below, we map each engine to its papers and explain how AEON applies the research.

🔎

Symbolic Execution

Correctness

AEON uses symbolic execution to explore all feasible paths through a function. Variables are treated as unknown symbols; at each branch, both sides are explored. Path conditions accumulate and are checked by the Z3 SMT solver. When a bug is found, AEON generates a concrete counterexample — the exact input values that trigger the failure.

  • 1976 King, J.C. “Symbolic Execution and Program Testing.” CACM
  • 2008 Cadar, C., Dunbar, D., & Engler, D. “KLEE: Unassisted and Automatic Generation of High-Coverage Tests for Complex Systems Programs.” OSDI
  • 2008 De Moura, L. & Bjørner, N. “Z3: An Efficient SMT Solver.” TACAS
🔐

Abstract Interpretation

Correctness

Tracks value ranges using Galois connections between concrete and abstract domains. AEON implements interval, sign, and congruence abstract domains with widening for convergence and narrowing for precision. Detects division-by-zero, integer overflow, and postcondition violations.

  • 1977 Cousot, P. & Cousot, R. “Abstract Interpretation: A Unified Lattice Model for Static Analysis of Programs by Construction or Approximation of Fixpoints.” POPL
  • 2019 Distefano, D. et al. “Scaling Static Analyses at Facebook.” CACM
📜

Hoare Logic / wp-Calculus

Correctness

Mechanically verifies function contracts (requires/ensures) by computing weakest preconditions and discharging verification conditions to the Z3 SMT solver. Loop invariants are inferred using Houdini’s algorithm.

  • 1969 Hoare, C.A.R. “An Axiomatic Basis for Computer Programming.” CACM
  • 1975 Dijkstra, E.W. “Guarded Commands, Nondeterminacy and Formal Derivation of Programs.” CACM
  • 2001 Flanagan, C. & Leino, K.R.M. “Houdini, an Annotation Assistant for ESC/Java.” FME
🎯

Liquid Types

Correctness

Refinement types annotate base types with logical predicates (e.g., {v: Int | v > 0}). AEON infers the strongest refinements via predicate abstraction over qualifier templates, solving with CEGAR.

  • 2008 Rondon, P.M., Kawaguchi, M., & Jhala, R. “Liquid Types.” PLDI
  • 2014 Vazou, N. et al. “Refinement Types for Haskell.” ICFP
  • 2018 Vazou, N. et al. “Refinement Reflection: Complete Verification with SMT.” POPL
🧮

Dependent Types / Curry-Howard

Correctness

Full Pi types where return types depend on input values. Via the Curry-Howard correspondence, types are propositions and programs are proofs. AEON implements bidirectional type checking with universe hierarchy.

  • 1984 Martin-Löf, P. Intuitionistic Type Theory. Bibliopolis.
  • 1988 Coquand, T. & Huet, G. “The Calculus of Constructions.” I&C
⏳

Size-Change Termination

Safety

Decides termination using Ramsey’s theorem. Builds size-change graphs at each call site, computes transitive closure, and verifies a strict decrease exists for every idempotent self-loop. Sound and complete for SCT.

  • 2001 Lee, C.S., Jones, N.D., & Ben-Amram, A.M. “The Size-Change Principle for Program Termination.” POPL
🚧

Separation Logic

Safety

Heap safety verification via spatial reasoning. The separating conjunction (P * Q) splits the heap into disjoint parts. The frame rule enables modular verification. Bi-abduction infers pre/postconditions automatically.

  • 2002 Reynolds, J.C. “Separation Logic: A Logic for Shared Mutable Data Structures.” LICS
  • 2011 Calcagno, C. et al. “Compositional Shape Analysis by Means of Bi-Abduction.” JACM
  • 2019 O’Hearn, P.W. “Incorrectness Logic.” POPL
🔄

Concurrency Verification

Safety

Race and deadlock detection combining lockset analysis, happens-before ordering, and lock-order cycle detection. Finds data races, deadlocks, atomicity violations, and unreleased locks.

  • 1976 Owicki, S. & Gries, D. “An Axiomatic Proof Technique for Parallel Programs.” Acta Informatica
  • 1978 Lamport, L. “Time, Clocks, and the Ordering of Events in a Distributed System.” CACM
  • 1997 Savage, S. et al. “Eraser: A Dynamic Data Race Detector for Multithreaded Programs.” TOCS
  • 2005 Flanagan, C. & Godefroid, P. “Dynamic Partial-Order Reduction for Model Checking Software.” POPL
🕷

Taint Analysis

Security

Tracks untrusted data from sources (user input, HTTP params) to sinks (SQL queries, HTML output, OS commands). Detects SQL injection, XSS, command injection, path traversal, and SSRF.

  • 2009 Tripp, O. et al. “TAJ: Effective Taint Analysis of Web Applications.” PLDI
  • 2010 Schwartz, E.J. et al. “All You Ever Wanted to Know About Dynamic Taint Analysis and Forward Symbolic Execution.” IEEE S&P
  • 2014 Arzt, S. et al. “FlowDroid: Precise Context, Flow, Field, Object-sensitive and Lifecycle-aware Taint Analysis for Android Apps.” PLDI
🔒

Information Flow / Noninterference

Security

Security type system proving noninterference: secret inputs cannot influence public outputs. Detects explicit flows, implicit flows (via control flow), and termination channels.

  • 1996 Volpano, D., Smith, G., & Irvine, C. “A Sound Type System for Secure Flow Analysis.” JCS
🔬

Bounded Model Checking

Safety

Unrolls programs to bound k, encodes as SAT/SMT, and checks temporal properties. Verifies assertions, safety properties, and loop invariants with counterexample traces.

  • 1986 Clarke, E.M., Emerson, E.A., & Sistla, A.P. “Automatic Verification of Finite-State Concurrent Systems Using Temporal Logic Specifications.” TOPLAS
  • 1999 Biere, A. et al. “Symbolic Model Checking without BDDs.” TACAS
  • 2009 Jhala, R. & Majumdar, R. “Software Model Checking.” ACM Computing Surveys
🗈

Shape Analysis

Safety

Verifies linked data structures using 3-valued logic (Kleene). Canonical abstraction merges concrete nodes into summary nodes. Proves list acyclicity, tree properties, and absence of null dereferences.

  • 2002 Sagiv, M., Reps, T., & Wilhelm, R. “Parametric Shape Analysis via 3-Valued Logic.” TOPLAS
⚡

Algebraic Effects

Architecture

Effects form a bounded lattice with row-polymorphic typing. AEON tracks, infers, and verifies side effects as first-class citizens of the type system.

  • 2009 Plotkin, G. & Pretnar, M. “Handlers of Algebraic Effects.” ESOP
  • 2017 Leijen, D. “Type Directed Compilation of Row-Typed Algebraic Effects.” POPL
📐

Category-Theoretic Semantics

Architecture

Provides precise denotational semantics: pure functions are CCC morphisms, tasks are Kleisli morphisms in a graded monad, and data types are initial algebras of polynomial functors.

  • 1991 Moggi, E. “Notions of Computation and Monads.” I&C
🛡

Certified Compilation

Architecture

CompCert-style simulation proofs verify each compiler pass preserves semantics. Forward simulation, invariant tracking, translation validation, and compositional correctness.

  • 2009 Leroy, X. “Formal Verification of a Realistic Compiler.” CACM
🔭

Hindley-Milner Type Inference

Correctness

Principal type inference via Algorithm W. Every well-typed expression has a most-general type inferred automatically. Let-polymorphism enables reuse across types. The occurs check prevents infinite cyclic types. OutsideIn(X) extends this to GHC-style modular inference with local assumptions.

  • 1969 Hindley, J.R. “The Principal Type-Scheme of an Object in Combinatory Logic.” TAMS
  • 1978 Milner, R. “A Theory of Type Polymorphism in Programming.” JCSS
  • 1982 Damas, L. & Milner, R. “Principal Type-Schemes for Functional Programs.” POPL
  • 2011 Vytiniotis, D. et al. “OutsideIn(X): Modular Type Inference with Local Assumptions.” JFP
🔒

Ownership & Borrow Checking

Safety

Affine type theory for memory safety without GC. At most ONE mutable reference or any number of shared references at a time. References cannot outlive their owned values. RustBelt provides the full semantic model in Iris (higher-order concurrent separation logic), proving Rust’s type system is sound.

  • 1990 Wadler, P. “Linear Types Can Change the World!” IFIP TC 2
  • 1997 Tofte, M. & Talpin, J.P. “Region-Based Memory Management.” I&C
  • 1998 Clarke, D. et al. “Ownership Types for Flexible Alias Protection.” OOPSLA
  • 2018 Jung, R. et al. “RustBelt: Securing the Foundations of the Rust Programming Language.” POPL
  • 2019 Weiss, A. et al. “Oxide: The Essence of Rust.” arXiv:1903.00982
⚖

Linear & Resource Logic

Correctness

Substructural type systems for precise resource accounting. Linear logic removes weakening and contraction — resources must be used exactly once. Quantitative Type Theory generalizes with usage annotations: 0 (erased), 1 (linear), ω (unrestricted). Enables safe in-place mutation and compile-time file/socket lifecycle enforcement.

  • 1987 Girard, J.Y. “Linear Logic.” TCS
  • 2018 Atkey, R. “Syntax and Semantics of Quantitative Type Theory.” LICS
  • 2018 Bernardy, J.P. et al. “Linear Haskell.” POPL
⚡

Algebraic Effect Handler Verification

Architecture

Verifies handler completeness, continuation linearity, and row type correctness. Detects non-commutative effect compositions where handler order changes semantics (State+Exception). Ensures the resume continuation is used the correct number of times for the declared handler style (linear, abort, or multi-shot).

  • 2013 Plotkin, G. & Pretnar, M. “Handling Algebraic Effects.” LMCS
  • 2015 Bauer, A. & Pretnar, M. “Programming with Algebraic Effects and Handlers.” JLAMP
  • 2017 Lindley, S. et al. “Do Be Do Be Do.” POPL
  • 2019 Biernacki, D. et al. “Abstracting Algebraic Effects.” POPL
🔄

Session Types

Safety

Verifies communication protocols for concurrent and distributed systems. Multiparty session types extend binary protocols to N-party global protocols projected to each participant. Via Curry-Howard, session types correspond to linear logic propositions — deadlock freedom follows from cut elimination.

  • 1998 Honda, K. et al. “Language Primitives for Structured Communication-Based Programming.” ESOP
  • 2008 Honda, K., Yoshida, N., & Carbone, M. “Multiparty Asynchronous Session Types.” POPL
  • 2012 Wadler, P. “Propositions as Sessions.” ICFP
🛠

Typestate Analysis

Safety

Object protocol enforcement — operations are only valid in the correct state. State machines track object lifecycles (e.g., Closed → open() → Open → close() → Closed). Detects use-before-open, use-after-close, and double-close. Aliasing-aware typestate tracks state through references.

  • 1986 Strom, R.E. & Yemini, S. “Typestate: A Programming Language Concept for Enhancing Software Reliability.” IEEE TSE
  • 2004 DeLine, R. & Fähndrich, M. “Typestates for Objects.” ECOOP
  • 2007 Bierhoff, K. & Aldrich, J. “Modular Typestate Checking of Aliased Objects.” OOPSLA
👁

Differential Privacy Verification

Security

Tracks privacy budgets via sensitivity typing. (ε,δ)-differential privacy guarantees output distributions change by at most eε when one record changes. Sensitivity types track amplification; linear dependent types enforce composition bounds. Detects privacy budget overruns and missing noise addition at compile time.

  • 2006 Dwork, C. et al. “Calibrating Noise to Sensitivity in Private Data Analysis.” TCC
  • 2010 Reed, J. & Pierce, B. “Distance Makes the Types Grow Stronger.” ICFP
  • 2013 Gaboardi, M. et al. “Linear Dependent Types for Differential Privacy.” POPL
🎲

Probabilistic Program Analysis

Correctness

Measure-theoretic verification of probabilistic programs. Programs are modeled as measure transformers over probability distributions. Martingale-based ranking functions prove almost-sure termination. Formal certification of cryptographic proofs via probabilistic relational Hoare logic (pRHL).

  • 1981 Kozen, D. “Semantics of Probabilistic Programs.” JCSS
  • 2009 Barthe, G. et al. “Formal Certification of Code-Based Cryptographic Proofs.” POPL
  • 2013 Chakarov, A. & Sankaranarayanan, S. “Probabilistic Program Analysis with Martingales.” CAV
  • 2014 Gordon, A. et al. “Probabilistic Programming.” ICSE

Coinductive Verification

Correctness

Verification of infinite structures and reactive systems via greatest fixpoints. Bisimulation proves behavioral equivalence of labeled transition systems. Final coalgebras give the universal property of observable behavior. Productivity checking ensures corecursive definitions are guarded. Up-to techniques enable efficient bisimulation proofs.

  • 1989 Milner, R. Communication and Concurrency. Prentice Hall.
  • 1997 Jacobs, B. & Rutten, J. “A Tutorial on (Co)Algebras and (Co)Induction.” EATCS Bulletin
  • 2012 Pous, D. & Sangiorgi, D. Enhancements of the Bisimulation Proof Method. Cambridge Tracts.
  • 2013 Abel, A. & Pientka, B. “Wellfounded Recursion with Copatterns.” ICFP
🔬

Quantum Program Verification

Correctness

Quantum Hoare logic verifies correctness over density matrices. A linear type system enforces the no-cloning theorem — qubits used more than once are a compile error. Categorical semantics (dagger compact categories) gives precise denotational meaning. Decoherence analysis flags circuits too deep for NISQ hardware.

  • 2004 Selinger, P. “Towards a Quantum Programming Language.” MSCS
  • 2004 Abramsky, S. & Coecke, B. “A Categorical Semantics of Quantum Protocols.” LICS
  • 2011 Ying, M. “Floyd-Hoare Logic for Quantum Programs.” TOPLAS
  • 2019 Zhou, L., Yu, N., & Ying, M. “An Applied Quantum Hoare Logic.” PLDI

Program Synthesis Verification

Correctness

Verifies that synthesized programs satisfy their specifications. CEGIS loops are checked for correctness. Sketch holes must be consistently filled. Refinement type synthesis produces programs correct by construction. DreamCoder’s library learning is verified for behavioral equivalence via wake-sleep Bayesian compression.

  • 2006 Solar-Lezama, A. et al. “Combinatorial Sketching for Finite Programs.” ASPLOS
  • 2011 Gulwani, S. “Automating String Processing in Spreadsheets Using Input-Output Examples.” POPL
  • 2013 Alur, R. et al. “Syntax-Guided Synthesis.” FMCAD
  • 2016 Polikarpova, N. et al. “Program Synthesis from Polymorphic Refinement Types.” PLDI
  • 2021 Ellis, K. et al. “DreamCoder: Bootstrapping Inductive Program Synthesis.” PLDI

Smart Contract Verification

Security

Formal security analysis for Solidity and EVM bytecode. Detects reentrancy (the DAO hack pattern), integer overflow/underflow, missing access control, and temporal safety violations. KEVM provides a complete formal semantics of the EVM. Securify uses compliance patterns; VerX uses reachability logic for temporal properties.

  • 2016 Bhargavan, K. et al. “Formal Verification of Smart Contracts.” PLAS
  • 2018 Hildenbrandt, E. et al. “KEVM: A Complete Formal Semantics of the Ethereum Virtual Machine.” CSF
  • 2018 Tsankov, P. et al. “Securify: Practical Security Analysis of Smart Contracts.” CCS
  • 2020 Permenev, A. et al. “VerX: Safety Verification of Smart Contracts.” IEEE S&P
🌐

WebAssembly Verification

Safety

Stack-based type checking verifies every WASM instruction in O(n) time. Memory bounds checking ensures all accesses stay within linear memory. Sandboxing proofs guarantee the module cannot escape its memory. Structured control flow prevents ROP attacks. The full WASM spec is mechanized in Isabelle/HOL.

  • 2017 Haas, A. et al. “Bringing the Web up to Speed with WebAssembly.” PLDI
  • 2018 Watt, C. “Mechanising and Verifying the WebAssembly Specification.” CPP
  • 2020 Lehmann, D. et al. “Everything Old is New Again: Binary Security of WebAssembly.” USENIX Security
🔐

Cryptographic Protocol Verification

Security

Formal security proofs under the Dolev-Yao attacker model. ProVerif uses Horn clause resolution to prove secrecy and authentication. Detects replay attacks (the Needham-Schroeder flaw), broken algorithms (MD5, SHA-1, DES, RC4), and weak key sizes per NIST SP 800-131A Rev 2.

  • 1983 Dolev, D. & Yao, A. “On the Security of Public Key Protocols.” IEEE TIT
  • 1996 Lowe, G. “Breaking and Fixing the Needham-Schroeder Public-Key Protocol.” TACAS
  • 2001 Blanchet, B. “An Efficient Cryptographic Protocol Verifier Based on Prolog Rules.” CSFW
  • 2001 Abadi, M. & Fournet, C. “Mobile Values, New Names, and Secure Communication.” POPL
🧠

ML-Assisted Memory Safety

Safety

Combines neural bug detection with classical Andersen pointer analysis. ProGraML represents programs as graphs with control, data, and call edges — a GNN learns to propagate dataflow facts. Neural models guide path selection; classical analysis verifies the result. Detects null dereference, use-after-free, buffer overflow, and double-free patterns.

  • 2018 Allamanis, M. et al. “A Survey of Machine Learning for Big Code and Naturalness.” ACM Computing Surveys
  • 2021 Cummins, C. et al. “ProGraML: A Graph-based Program Representation for Data Flow Analysis.” ICML
🤖

Neurosymbolic Proof Assistance

Correctness

AI-guided formal proof search with classical verification kernels. A language model trained on (proof state, tactic) pairs suggests next steps; each step is verified by a proof kernel. Loop invariants are synthesized by a neural model and verified classically. Differentiable proving uses soft unification via embedding similarity.

  • 2017 Rocktäschel, T. & Riedel, S. “End-to-end Differentiable Proving.” NeurIPS
  • 2020 Polu, S. & Han, J.M. “Generative Language Modeling for Automated Theorem Proving.” arXiv:2009.03393
  • 2020 Lample, G. & Charton, F. “Deep Learning for Symbolic Mathematics.” ICLR
  • 2022 Han, J.M. et al. “Proof Artifact Co-Training.” ICLR

Engines 32–40: Applied Verification

The remaining engines apply foundational techniques to specific, practical verification domains.

🔀

Gradual Typing

Correctness

Seamless migration between dynamic and static typing with the gradual guarantee. Blame tracking pinpoints which boundary caused a runtime type error. Cast insertion is sound by construction.

  • 2006 Siek, J. & Taha, W. “Gradual Typing for Functional Languages.” Scheme Workshop
  • 2016 Garcia, R. et al. “Abstracting Gradual Typing.” POPL
⏲

Complexity Analysis

Correctness

Automatic worst-case resource bound inference via amortized potential functions. Resource Aware ML assigns potential to data structures; each operation consumes or transfers potential. Derives tight O(n), O(n log n), O(n²) bounds automatically.

  • 2012 Hoffmann, J. et al. “Multivariate Amortized Resource Analysis.” TOPLAS
📋

Relational Verification

Correctness

Proves properties relating two program executions — equivalence, non-interference, and compiler correctness. Product programs reduce relational reasoning to standard Hoare logic. Relational Hoare logic enables direct two-execution proofs.

  • 2004 Benton, N. “Simple Relational Correctness Proofs for Static Analyses and Program Transformations.” POPL
  • 2011 Barthe, G. et al. “Relational Verification Using Product Programs.” FM
🔗

API Contract Verification

Safety

Verifies REST/gRPC contracts against OpenAPI 3.0 schemas. Checks request/response types, required fields, status code coverage, and rate limit annotations. Catches schema drift between implementation and specification.

🗑

Dead Code & Reachability

Architecture

CFG reachability analysis detects unreachable branches, unused functions, and dead stores. Tree-shaking verification ensures no live code is removed. Detects impossible conditions and redundant checks.

🚫

Null Safety

Safety

Flow-sensitive nullable type analysis based on Hoare’s “billion dollar mistake.” A three-point lattice (NonNull ≤ Nullable ≤ Null) tracks nullability through branches, assignments, and returns. Guards narrow types; dereferences on Nullable values are flagged.

🔢

Numeric Safety

Safety

IEEE 754 floating-point precision analysis. Detects catastrophic cancellation (subtracting nearly-equal values), accumulation drift in summation loops, and precision loss from widening conversions. Based on Goldberg’s foundational floating-point survey.

  • 1991 Goldberg, D. “What Every Computer Scientist Should Know About Floating-Point Arithmetic.” ACM Computing Surveys
⚠

Error Handling Completeness

Safety

Verifies that every error path is handled. Tracks exception propagation through call chains. Detects unchecked exceptions, swallowed errors, and missing error branches. Railway-oriented programming patterns ensure exhaustive error coverage.

🛠

Interpolation & Predicate Abstraction

Correctness

Craig interpolation extracts relevant predicates from infeasible counterexample paths. CEGAR loop refines the abstraction until a real bug is found or the property is proved. Powers the SLAM model checker and AEON’s automatic invariant generation.

  • 1957 Craig, W. “Three Uses of the Herbrand-Gentzen Theorem.” JSL
  • 2002 Ball, T. & Rajamani, S.K. “The SLAM Project: Debugging System Software via Static Analysis.” POPL
